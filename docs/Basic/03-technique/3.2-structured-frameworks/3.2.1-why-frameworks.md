# 3.2.1 为什么需要框架

## 经过本节学习，你将掌握

- 理解框架的核心价值：减少思考负担，提高输出稳定性
- 明确框架不是限制，而是起点
- 知道什么时候用框架，什么时候灵活发挥
- 理解故事化 Prompt 与结构化框架的适用场景差异


## 一个真实的困扰

小李学完 3.1 节后，知道了要给 AI 提供上下文。但每次写提示词时，他还是会陷入这样的状态：

> "我知道要告诉 AI 项目背景...等等，还有什么来着？技术栈、约束条件、输出格式...每次都要从头想一遍，好累。"

这不是小李的问题，而是**认知负担**的问题。

人脑的工作记忆容量有限。当你同时要思考"说什么"和"怎么说"时，很容易顾此失彼——要么遗漏关键信息，要么表述混乱。

框架的价值，就是**把"怎么说"变成固定格式，让你专注于"说什么"**。


## 框架的三个核心价值

### 1. 减少思考负担

没有框架时，你需要每次从零思考：
- 该说什么？
- 先说什么后说什么？
- 有没有遗漏？

有了框架后，你只需要**填空**：
- Situation（情境）→ 填入项目背景
- Challenge（挑战）→ 填入当前问题
- ...以此类推

框架把"开放式问题"变成了"填空题"，认知负担大幅降低。

### 2. 提高输出稳定性

斯坦福大学人机交互研究院的研究表明：使用结构化框架的提示词，任务完成准确率比随机提问高出 30% 以上。

原因很简单：框架确保了**关键信息不遗漏**。就像飞行员起飞前的检查清单，不是因为他们记不住，而是因为清单能避免人为疏忽。

### 3. 降低沟通成本

当你和 AI 建立了固定的"沟通协议"，AI 能更快理解你的意图。

这就像和老同事合作：你们有默契的沟通方式，不需要每次都从头解释。框架就是你和 AI 之间的"默契"。


## 框架不是限制，而是起点

你可能会担心：用框架会不会太死板？

答案是：**框架是起点，不是终点**。

| 阶段 | 做法 |
|-----|------|
| 初学阶段 | 严格按框架填写，确保不遗漏 |
| 熟练阶段 | 根据任务复杂度简化或扩展 |
| 高手阶段 | 框架内化成直觉，自然地组织信息 |

就像学开车：一开始要死记"打灯、看镜、转向"的顺序，熟练后这些动作会变成本能。


## 什么时候用框架，什么时候自由发挥

| 场景 | 建议 | 原因 |
|-----|------|------|
| 复杂的技术任务 | 用完整框架 | 防止遗漏关键约束 |
| 简单的快速提问 | 用精简框架或自由发挥 | 节省时间 |
| 第一次做某类任务 | 用完整框架 | 帮助你理清思路 |
| 重复做过的任务 | 可以简化 | 你已经知道关键要素 |
| AI 输出不符合预期 | 回到框架检查 | 定位遗漏了什么信息 |

**一个实用建议**：当你不确定时，用框架；当框架感觉多余时，简化它。


## 故事化 Prompt vs 结构化框架

第二章介绍的故事化 Prompt 和本节的结构化框架，各有适用场景：

| 对比维度 | 故事化 Prompt | 结构化框架 |
|---------|--------------|-----------|
| **核心驱动** | 情感和痛点 | 逻辑和约束 |
| **表达方式** | 叙事性，像讲故事 | 列表式，像填表格 |
| **适合任务** | 产品设计、用户需求沟通 | 代码实现、技术任务 |
| **AI 理解重点** | 用户是谁、为什么需要 | 怎么做、有什么限制 |
| **典型场景** | "帮我设计一个待办清单" | "帮我实现添加任务功能" |

**实际使用中，两者可以组合**：

1. 先用故事化 Prompt 描述整体需求（产品层面）
2. 再用结构化框架拆解具体任务（技术层面）

例如：
- 第一轮对话：用故事化 Prompt 描述"我是谁、我的痛点、我想要什么"
- 第二轮对话：用 S.C.A.F.F. 框架描述"当前要实现的具体功能"


## 本节要点

✓ **框架的价值**：减少思考负担、提高输出稳定性、降低沟通成本

✓ **框架不是限制**：初学时严格遵循，熟练后灵活调整

✓ **选择原则**：复杂任务用完整框架，简单任务可以简化

✓ **与故事化 Prompt 的关系**：情感驱动 vs 逻辑驱动，可以组合使用
