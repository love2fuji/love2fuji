# 2.8.2 开发前检查清单

在你打开AI工具、写第一行Prompt之前，用这份清单做最后的检查。


## 60秒快速版：红绿灯判断法

没有时间做完整检查？用这三个问题快速判断：

| # | 问题 | 你的回答 |
|---|------|---------|
| 1 | 这个东西是给谁用的？（能说出具体的人或自己） | |
| 2 | 它解决什么问题？（能描述一个具体的烦恼场景） | |
| 3 | 第一版只做什么？（能用一句话说清楚） | |

**判断标准：**

- 🟢 **三个问题都能一句话回答** → 可以开始
- 🟡 **有的问题说不清楚** → 需要再想想，回顾对应章节
- 🔴 **有的问题完全答不上来** → 先停下来，从2.1节重新开始


## 5分钟完整版：分维度检查

如果你有5分钟，用这份更完整的清单：

### 问题定义（30秒）

```markdown
□ 我知道这个问题来自哪里（自己的烦恼/他人的抱怨/工作流断点）
□ 这个问题重复出现，不是一次性的
□ 我能用一句话描述这个问题
```

### 用户理解（60秒）

```markdown
□ 我知道用户是谁（具体到能叫出名字或描述特征）
□ 我知道用户现在怎么解决这个问题（替代方案）
□ 我理解用户的情绪（焦虑？烦躁？无奈？）
□ 我知道用户会在什么场景下使用
```

### 风险预判（60秒）

```markdown
□ 我想过什么情况下会失败
□ 我有一个"避坑清单"或预防措施
□ 我知道这个项目的边界（什么不做）
```

### MVP范围（60秒）

```markdown
□ 我的核心假设是什么（一句话）
□ 我知道怎么验证这个假设（什么标准算成功）
□ 第一版的核心功能不超过3个
□ 我有一个明确的"不做清单"
```

### 最终确认（30秒）

```markdown
□ 我能用灵魂三问的格式总结：
  - 用户是谁：_______________
  - 痛点在哪：_______________
  - 为什么选我/为什么值得做：_______________
```


## 问题诊断表

如果你发现自己卡住了，对照这张表找到问题：

| 你的状态 | 可能的原因 | 建议行动 |
|---------|-----------|---------|
| 不知道做什么项目 | 没有系统地观察生活中的问题 | 回到2.7节，用"烦恼日记"记录一周 |
| 想做的功能太多，不知道砍哪个 | 没有明确核心假设 | 回到2.3节，问自己"第一版要验证什么" |
| 说不清楚用户是谁 | 用户画像太模糊 | 回到2.4节，用三维画像法重新描述 |
| 担心做出来没人用 | 没有做风险预判 | 回到2.2节，做一次Pre-mortem分析 |
| 解释不清楚这个东西有什么用 | 任务定义不清晰 | 回到2.1节，用JTBD模板重新描述 |
| 不知道自己的项目和别人有什么区别 | 没想清楚差异化 | 回到2.5节，回答"为什么选你" |
| 场景特殊，不知道怎么调整 | 没有场景化思考 | 回到2.6节，找到对应场景的侧重点 |

## 不同场景的简化检查

并非所有项目都需要完整检查。根据你的场景，选择合适的深度：

| 场景 | 必须检查 | 可选检查 |
|-----|---------|---------|
| **给自己用的小工具** | 问题定义 + MVP范围 | 用户理解（自己就是用户） |
| **数据分析/报告** | 问题定义 + 用户理解（谁看） | 风险预判 |
| **自动化脚本** | 问题定义 + 风险预判（容错性） | 用户理解 |
| **给家人/朋友做** | 用户理解 + MVP范围 | 风险预判 |
| **想给很多人用的产品** | 全部检查 | 无 |


## 本节要点

✓ **60秒版是底线**：无论多急，三个问题必须能回答。

✓ **诊断表帮你定位问题**：卡住了不要硬冲，找到问题根源再继续。

✓ **不同场景深度不同**：给自己用的工具不需要完整的用户画像，但核心假设必须清楚。
