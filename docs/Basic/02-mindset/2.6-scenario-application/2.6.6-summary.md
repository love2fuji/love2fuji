# 2.6.6 本节总结：场景化思维速查表

## 核心洞见回顾

本节的核心观点是：

> **前面学的思维工具不是「产品经理专用」，而是「定义问题专用」。不管你想做什么，核心思维方式都是一样的。**


## 场景速查表

| 场景 | 核心问题 | 首要思维工具 | 常见陷阱 |
|-----|---------|------------|---------|
| **数据分析** | 这个分析要回答什么业务问题？ | JTBD思维 | 做了50张图但不知道结论是什么 |
| **自动化脚本** | 这个任务值得花时间自动化吗？ | 逆向思维+减法思维 | 一次性想自动化整个流程 |
| **个人工具** | 我的真实痛点是什么？ | 减法思维 | 过度设计，做完就不用 |
| **家人工具** | 他们会在什么场景下怎么用？ | 故事思维 | 按自己的习惯设计，家人用不了 |


## 思维工具 × 场景 应用矩阵

| 思维工具 | 数据分析 | 自动化脚本 | 个人工具 | 家人工具 |
|---------|---------|-----------|---------|---------|
| **JTBD思维** | 谁要看？要做什么决策？ | 这个任务帮我完成什么？ | 我要完成什么任务？ | 他们要完成什么任务？ |
| **逆向思维** | 什么会让分析没价值？ | 什么情况下脚本会出错？ | 什么会让我做完就不用？ | 什么会让他们用不了？ |
| **减法思维** | 最核心的3个问题是什么？ | 先自动化最痛的那一步 | 第一版只做1个功能 | 只保留他们需要的功能 |
| **故事思维** | 谁在什么场景下看这个分析？ | 什么时候运行？结果存哪？ | 我什么时候用？用什么设备？ | 他们什么时候用？怎么打开？ |
| **灵魂三问** | 谁？关心什么？为什么有价值？ | 谁用？什么最痛？为什么值得做？ | 我是谁？痛点是什么？ | 他们是谁？真实痛点是什么？ |


## 通用 Prompt 框架

不管你做什么场景，这个框架都适用：

```markdown
## 背景
- 我是：[你的角色/身份]
- 我要做：[一句话描述目标]
- 给谁用/谁会看：[用户/受众]
- 使用场景：[什么时候、在哪、用什么设备]

## 核心问题/任务
[用一句话描述要解决的核心问题或完成的核心任务]

## 详细说明
[根据场景填写具体信息：数据说明/当前流程/功能需求等]

## 第一版只需要
[列出1-3个核心功能/核心输出]

## 不需要
[明确说不做什么，避免AI过度发挥]

## 可能出错的情况
[如果适用，列出异常情况及期望的处理方式]
```


## 开始新项目前的5个问题

在开始任何项目之前，花5分钟问自己这5个问题：

| # | 问题 | 如果答不上来 |
|---|------|------------|
| 1 | 这个东西要帮谁完成什么任务？ | 任务不明确，先想清楚再动手 |
| 2 | 我/用户会在什么场景下使用？ | 场景不清，设计出来可能没人用 |
| 3 | 什么情况下会失败/出错/用不了？ | 没想过这个问题，容易踩坑 |
| 4 | 第一版最核心的1个功能是什么？ | 想做的太多，可能做不完 |
| 5 | 我能用1句话向别人解释这是什么吗？ | 如果说不清，可能自己也没想清楚 |

**5个问题都能回答，就可以开始了。**


## 场景选择决策树

```
我有一个想法，想用AI帮我做
    ↓
这是什么类型的任务？

→ 我有一堆数据，想看出点东西 → 数据分析场景（2.6.2）
→ 我有一个重复做的事情，想让它自动 → 自动化场景（2.6.3）
→ 我想做一个给自己用的小工具 → 个人工具场景（2.6.4）
→ 我想做一个给家人/朋友用的东西 → 家人工具场景（2.6.5）
→ 我想做一个给很多人用的产品 → 回到2.1-2.5，走完整的产品思维流程
```


## 本节核心要点

✓ **思维工具是通用的**——不管你做数据分析、自动化脚本还是小工具，核心思维方式都是「先想清楚问题」

✓ **场景不同，侧重不同**——数据分析侧重JTBD（回答什么问题），自动化侧重逆向思维（什么会出错），个人工具侧重减法思维（避免过度设计），家人工具侧重故事思维（站在他们角度）

✓ **Prompt质量取决于问题定义**——AI是一个超级执行者，但「做什么」只有你能决定。问题定义得越清楚，AI的输出越有用

✓ **先做最小版本**——不管什么场景，第一版都只做最核心的1-2个功能，验证了再加


## 下一章预告

你现在知道了如何在不同场景中应用思维工具。但还有一个问题：

**我怎么知道该做什么？**

很多人卡在「我不知道做什么项目」——其实不是没有问题，而是没学会发现问题。
