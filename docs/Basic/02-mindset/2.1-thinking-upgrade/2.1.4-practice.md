# 2.1.4 实践练习：用JTBD重新审视你的想法

理论学完了，现在轮到你动手了。

这个练习会帮你把刚学到的JTBD思维应用到你自己的想法上。


## 练习一：填写任务描述模板

拿出你现在想做的一个项目（或者用下面的示例），完成这个模板：

### 模板

```markdown
### 我的原始想法
我想做一个 _______________________

### JTBD任务描述

**用户是谁？**
[描述具体的人，不是"所有人"]

**情境是什么？**
[什么时候、什么情况下会用到？]

**要完成什么任务？**
[用户真正想达成的目标是什么？]

**期望的结果/感受是什么？**
[功能层面 + 情感层面]

**目前的替代方案？**
[用户现在怎么解决这个问题？]

**替代方案的痛点？**
[现有方案哪里不好用？]

### 重新表述

当 [某类用户] 在 [某个情境] 下，
他们想要 [完成某个任务]，
以便于 [获得某种结果/感受]。

目前他们的方案是 [现有解决方式]，
但这个方案的问题是 [痛点]。
```

### 填写示例

```markdown
### 我的原始想法
我想做一个读书笔记工具

### JTBD任务描述

**用户是谁？**
上班族，每天通勤时在手机上看电子书，每周能看1-2本

**情境是什么？**
看书时遇到好句子想记下来，或者读完后想回顾要点

**要完成什么任务？**
保存书中的精华内容，日后能快速找到和回顾

**期望的结果/感受是什么？**
功能：读完的书有沉淀，不是看完就忘
情感：觉得时间没白花，有成就感

**目前的替代方案？**
微信收藏、手机备忘录、读书App自带的划线功能

**替代方案的痛点？**
微信收藏找不到、备忘录没分类、App划线导出麻烦

### 重新表述

当上班族在通勤读书时看到好句子，
他们想要快速保存下来，并在日后能方便地回顾，
以便于让读书有沉淀、感到时间没白花。

目前他们用微信收藏或备忘录，
但内容零散、难以检索、无法形成体系。
```


## 练习二：自检清单

完成练习一后，用这个清单检验你的任务描述是否合格：

- [ ] **用户具体吗？** 不是"所有人"，而是有具体特征的人
- [ ] **情境明确吗？** 说清楚了"什么时候、什么情况下"
- [ ] **任务是动词吗？** 是"完成某事"而不是"拥有某功能"
- [ ] **包含情感层面吗？** 不只是功能需求，还有心理需求
- [ ] **知道替代方案吗？** 了解用户现在怎么解决
- [ ] **痛点真实吗？** 替代方案的问题是你观察到的，不是想象的

**如果有任何一项打不上勾，回去继续完善你的描述。**


## 练习三：案例对比分析

看看下面两个产品，思考它们成功/失败的原因：

### 案例A：某习惯打卡App（失败）

**产品功能：**
- 21天习惯养成计划
- 每日打卡提醒
- 社交分享、排行榜
- 习惯统计、成就系统
- 付费解锁高级功能

**结果：**
上线3个月后，日活不到1000，最终下架

**问题分析：**
这个App假设用户需要"养成习惯"，但没有问：
- 用户真的想养成什么习惯？
- 为什么他们之前没养成？
- 一个App真的能解决意志力问题吗？

**JTBD视角：**
用户的任务不是"打卡"，而是"变成更好的自己"。
打卡只是手段，不是目的。
这个App把手段当成了目的。


### 案例B：一个简单的计数器网页（成功）

**产品功能：**
- 一个大按钮
- 点一下加一
- 显示今天的次数
- 没了

**结果：**
一位健身教练用它记录每天做了多少个俯卧撑。
没有花哨功能，但每天都在用。

**成功原因：**
精准解决了一个具体任务——"记录今天做了多少个"。
不多也不少。

**JTBD视角：**

```
当健身教练锻炼时，
他想要快速记录每组做了多少个，
以便于追踪进步、保持动力。

目前他用脑子记或者纸笔，
但容易忘记、事后补记不准确。
```


### 思考题

1. 案例A的功能比案例B多得多，为什么反而失败了？
2. 如果你是案例A的产品经理，用JTBD视角重新定义问题，你会怎么做？
3. 案例B成功的关键是什么？


## 练习输出

完成上面的练习后，你应该得到：

1. **一份完整的任务描述**：用JTBD模板重新定义你的想法
2. **通过自检的确认**：六项检查全部打勾
3. **对案例的思考**：理解为什么简单往往比复杂更好
