# 4.1.1 回顾第二章成果：小李的待办清单

> **本小节目标**：用 5 分钟把第二章的思考成果"打包"成可以交给 AI 的任务

## 从"思考"到"执行"

还记得第二章的小李吗？他是一个职场新人，经常忘事，想做一个待办清单来管理自己的任务。

在第二章，我们帮小李做了很多思考工作：用户画像、灵魂三问、MVP 定义、风险预判......

现在，这些思考要**真正变成代码**了。

但在开始写 Prompt 之前，我们需要把这些思考成果"打包"成一个简洁的"开发任务卡"。

## 小李案例的一页纸总结

如果你跟着第二章做了思考，现在把它整理成这样的格式：

### 灵魂三问

| 问题 | 答案 |
|------|------|
| **用户是谁** | 小李，职场新人，25岁，经常忘事 |
| **痛点在哪** | 脑子里装太多事，容易遗漏重要任务，事后懊悔 |
| **为什么选这个方案** | 简单、快速、随时能用，不需要学习成本 |

### MVP 功能范围

| 优先级 | 功能 | 说明 | 本章实现 |
|--------|------|------|---------|
| **P0** | 添加任务 | 输入任务名称，点击添加 | ✅ |
| **P0** | 显示列表 | 看到所有待办任务 | ✅ |
| **P0** | 标记完成 | 点击任务，标记为已完成 | ✅ |
| **P0** | 删除任务 | 删除不需要的任务 | ✅ |
| **P1** | 数据保存 | 刷新页面后数据还在 | ✅ |
| P2 | 分类标签 | 按类型给任务分组 | ❌ 留给你扩展 |
| P2 | 截止日期 | 设置任务的截止时间 | ❌ 留给你扩展 |
| P2 | 优先级 | 区分任务的紧急程度 | ❌ 留给你扩展 |

### 不做清单

第一版明确**不做**的事情：
- ❌ 多用户/账号系统
- ❌ 数据同步/云存储
- ❌ 移动端专属适配
- ❌ 提醒/通知功能

## 为什么现在不需要完美的 PRD

你可能会想："第二章不是教我们写 PRD 吗？我是不是应该把 PRD 写得更完整？"

不需要。

**第一版的目标是验证想法，不是做出完美产品。**

以小李的待办清单为例：
- 核心假设是"一个简单的待办清单能帮小李减少遗忘"
- 验证这个假设，只需要：能添加、能看到、能完成、能删除
- 其他功能都是在假设验证成功后才需要考虑的

所以，你现在手里的"灵魂三问 + MVP 功能列表"就够用了。

::: info 💡 提示
如果你在第二章写了更详细的 PRD，那更好。但如果你只有上面这些内容，完全足够开始了。
:::

## 把思考成果转化为"开发任务卡"

现在，把你的思考成果整理成这张"项目启动卡"：

```markdown
# 项目启动卡

## 项目名称
个人待办清单

## 一句话描述
帮助小李管理日常任务，减少遗忘的简单网页工具

## 目标用户
职场新人，经常忘事，需要一个随时能用的任务管理工具

## 第一版核心功能（按优先级）
1. 添加任务（输入任务名称，点击添加按钮）
2. 显示任务列表（看到所有待办任务）
3. 标记完成（点击任务，标记为已完成，视觉上有区分）
4. 删除任务（删除不需要的任务）
5. 数据保存（刷新页面后数据还在）

## 明确不做
- 账号系统
- 云同步
- 提醒通知
- 移动端专属适配

## 技术约束
- 纯前端实现（HTML + CSS + JavaScript）
- 使用浏览器本地存储（localStorage）保存数据
- 不需要后端服务器
```

这张卡片包含了你需要告诉 AI 的所有核心信息。在 4.2 节，我们会用第三章学的 S.C.A.F.F. 框架，把这些信息组织成 Prompt。

## 如果你有自己的项目想法

到目前为止，我们一直在用小李的待办清单作为案例。如果你想做自己的项目，完全可以。

**但建议第一次先跟着小李案例走完整个流程**，原因是：

1. 待办清单涵盖了最常用的操作（增删改查）
2. 遇到问题时，你可以对照教程排查
3. 完成后你会获得完整的经验，做自己的项目时更有信心

做完小李的项目后，你可以立刻用同样的方法做自己的项目。

## 本小节要点

✓ **你已经有了"需求文档"**：第二章的思考成果就是你的需求文档，现在只是把它整理成可执行的格式。

✓ **不需要完美**：灵魂三问 + MVP 功能列表就够开始了。

✓ **项目启动卡是你的指南针**：接下来的开发过程中，随时可以回来看这张卡，确认自己在做对的事情。
