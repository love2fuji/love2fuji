# 1.5.4 常见问题 FAQ

做第一个项目的时候，你可能会遇到一些困惑。

别担心，这些问题大家都会遇到。


## Q1："我的结果和教程不一样怎么办？"

**这太正常了。**

AI 每次生成的内容都会有些不同。就像你问两个设计师"做一个好看的页面"，他们给你的方案肯定也不一样。

::: tip 关键是
只要你的页面：
- 能正常显示
- 有名言和作者
- 看起来还不错

就算成功了！
:::

如果你想要和教程一模一样的效果，可以在 Prompt 里描述得更具体，比如：
- 指定具体的颜色（"背景用 #f0f4f8"）
- 指定具体的字体大小（"名言用 32px 的字体"）
- 指定具体的布局（"名言在页面垂直居中"）


## Q2："代码看不懂正常吗？"

**完全正常。**

起步阶段，你不需要看懂每一行代码。

就像你用微信发消息，不需要懂微信是怎么写出来的。

Vibe Coding 的核心理念是：**让 AI 处理代码语法，你专注于需求和结果**。

::: info 好消息
随着使用次数增多，你会自然地开始认识一些常见的代码结构。

如果你好奇某段代码是什么意思，可以问 AI：

```
请解释一下这段代码是做什么的，用最简单的话说
```

AI 会用你能懂的语言给你解释。这不是必须的，但会让你和 AI 的协作更顺畅。
:::


## Q3："可以用中文吗？"

**当然可以。**

现在的 AI 对中文的理解已经非常好了。

用中文写 Prompt，AI 完全能懂。

::: tip 什么时候用英文？
有些技术名词用英文会更准确，比如：
- "按钮" vs "button"
- "悬停效果" vs "hover effect"

但大部分情况下，用中文完全没问题。

如果你英文好，用英文也可以——效果差不多。
:::


## Q4："AI 生成的东西报错了怎么办？"

**把错误信息复制给 AI。**

这是最简单有效的方法：

```
页面报错了，错误信息是：
[粘贴错误信息]

请帮我修复
```

AI 会分析错误原因，然后给你修复后的代码。

::: warning 如果反复修不好
1. **简化你的需求**：把复杂功能拆成更小的步骤，一次只做一件事
2. **换一种描述方式**：同样的需求，用不同的话说，AI 可能更容易理解
3. **从最简版本重新开始**：删掉复杂功能，先让基础版跑起来，再逐步加功能

问题通常出在"描述不够清楚"，而不是工具不行。
:::


## Q5："这个网页怎么发给别人看？"

现在你做的网页只在工具的预览窗口里。想让别人访问，需要"部署"——这个我们在后面的章节会详细讲。

**现在不用着急，先把东西做出来再说。**


## Q6："我想保存这个项目怎么办？"

不同类型的工具保存方式不同：

| 工具类型 | 保存方式 |
|----------|----------|
| 网页版工具（如 Bolt.new、豆包） | 注册账号后自动保存，也可以导出代码 |
| AI 代码编辑器（如 Cursor、Trae） | 本地文件，随时保存 |

如果你只是想保留代码，最简单的方法是：
1. 复制 AI 生成的所有代码
2. 粘贴到一个文本文件里
3. 保存到电脑上


## Q7："我可以用这个做商业项目吗？"

可以，但建议分阶段：

1. **先用 Vibe Coding 验证想法**——快速做出原型，看看是否可行
2. **如果要正式上线**——涉及用户数据、支付等敏感功能时，需要学习进阶篇的安全（第6章）和测试（第9章）内容

很多成功的产品，都是从一个粗糙的原型开始的。本教程的进阶篇会教你如何把原型变成正式产品。


## Q8："不同工具界面不一样怎么办？"

**不用担心，核心操作都一样。**

无论你用 Bolt.new、豆包、Cursor 还是其他工具，基本流程都是：

1. 找到输入框
2. 用中文描述你想要什么
3. 等 AI 生成
4. 看预览效果
5. 不满意就继续说

界面按钮的位置可能不同，但"用自然语言描述需求"这个核心不会变。

如果找不到某个功能，直接问 AI："怎么在 [工具名] 里保存项目？"


## 还有其他问题？

如果你遇到了上面没提到的问题，试试这个方法：

**直接问 AI。**

```
我在用 [工具名] 做一个网页，遇到了这个问题：
[描述你的问题]

请问怎么解决？
```

AI 能帮你思考大部分问题——但记得验证它给的答案是否靠谱。

这也是 Vibe Coding 的精髓：**遇到问题，先问 AI，再自己验证**。
